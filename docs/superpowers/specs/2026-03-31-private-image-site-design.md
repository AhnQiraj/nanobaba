# 私人生图网站设计文档

## 1. 目标

为用户的妻子提供一个部署在 VPS 上的中文网页，用于通过已有 Gemini 中转站生成图片。网页只服务单个家庭用户，重点是简单、稳定、可直接访问，不要求复杂账号体系或多人协作能力。

本项目的核心目标：

- 提供单密码登录保护，避免页面对外裸奔
- 通过服务端代理调用 Gemini 生图模型，前端不暴露 API Key
- 在首页完成提示词输入、图片生成、结果查看、下载和历史浏览
- 历史记录保留 30 天，自动清理旧数据
- 支持手机和电脑访问
- 中转站地址、密钥、模型名全部通过 `.env` 配置，便于以后替换

不在本次范围内的内容：

- 多用户账号系统
- 支付、额度、计费
- 风格模板、复杂参数面板
- 图片编辑、局部重绘、对话式多轮创作
- 云对象存储

## 2. 推荐方案

采用 `Next.js + shadcn/ui + Tailwind CSS + SQLite`。

选择理由：

- `Next.js` 适合小型一体化应用，页面、API、鉴权逻辑可以放在一个项目中
- `shadcn/ui + Tailwind CSS` 足够做出清晰、轻量、适合中文场景的界面，且后续可调整性高
- `SQLite` 足够支撑单用户历史记录，不需要引入额外数据库服务
- 服务端代理模式能保证 `base_url`、`api_key`、`model` 全部留在服务器端

## 3. 系统架构

系统由 4 个核心部分组成：

### 3.1 Web 界面

用户通过浏览器访问网页。

页面包括：

- 登录页 `/login`
- 主页面 `/`

主页面集成：

- 提示词输入区
- 生成按钮和状态提示
- 当前结果图片区
- 最近历史列表

### 3.2 服务端业务层

Next.js 服务端负责：

- 校验登录态
- 读取环境变量
- 调用 Gemini 中转站
- 将返回的 base64 图片落盘
- 写入和读取 SQLite 历史记录
- 执行 30 天清理逻辑

### 3.3 数据存储

使用 SQLite 保存元数据，字段至少包括：

- `id`
- `prompt`
- `model`
- `image_path`
- `mime_type`
- `status`
- `error_message`
- `created_at`

状态字段用于区分：

- `success`
- `failed`

### 3.4 文件存储

图片文件保存在 VPS 本地目录，例如：

- `data/images/YYYY/MM/<id>.jpg`

不把完整 base64 直接存入数据库，原因：

- SQLite 文件不会无意义膨胀
- 文件下载路径更直接
- 清理旧记录时容易同步删除图片文件

## 4. 环境变量设计

以下配置全部通过 `.env` 提供：

- `APP_PASSWORD`
- `SESSION_SECRET`
- `GEMINI_PROXY_BASE_URL`
- `GEMINI_PROXY_API_KEY`
- `GEMINI_IMAGE_MODEL`
- `DATABASE_URL`
- `IMAGE_STORAGE_DIR`

建议含义如下：

- `APP_PASSWORD`：登录密码
- `SESSION_SECRET`：签名登录会话 cookie
- `GEMINI_PROXY_BASE_URL`：如 `https://mytoken.online`
- `GEMINI_PROXY_API_KEY`：中转站 API Key
- `GEMINI_IMAGE_MODEL`：默认模型，如 `gemini-3.1-flash-image`
- `DATABASE_URL`：SQLite 数据库文件路径
- `IMAGE_STORAGE_DIR`：图片落盘目录

代码中不写死任何中转站地址、密钥或模型名。服务端拼接请求地址时，使用：

- `${GEMINI_PROXY_BASE_URL}/v1/chat/completions`

这样以后更换中转站或模型时，只需调整 `.env`。

## 5. 鉴权设计

鉴权采用单密码登录 + 服务端 session。

流程如下：

1. 用户访问 `/login`
2. 输入密码并提交
3. 服务端将输入密码与 `APP_PASSWORD` 比较
4. 成功后创建服务端签名 cookie
5. 后续访问主页面和 API 接口时校验 cookie

要求：

- cookie 使用 `httpOnly`
- cookie 使用 `sameSite=lax`
- 在 HTTPS 场景下启用 `secure`
- 未登录时不允许访问生成接口与历史接口

这个方案满足单用户私有站需求，不引入复杂账号系统。

## 6. 页面设计

### 6.1 登录页 `/login`

元素尽量少：

- 标题
- 密码输入框
- 登录按钮
- 简单错误提示

目标是减少理解成本，不出现技术术语。

### 6.2 主页面 `/`

主页面包含 3 个核心区域：

#### A. 提示词输入区

- 大文本框
- 生成按钮
- 清空按钮
- 生成状态提示

用途是让用户专注于描述画面，不暴露复杂模型参数。

#### B. 当前结果区

- 显示最新生成图片
- 提供下载按钮
- 可选“再来一张”按钮

结果区需要足够大，因为这是页面核心。

#### C. 最近历史区

- 展示最近 30 天内生成成功的图片记录
- 每条包含缩略图、简短提示词摘要、时间
- 点击可重新查看图片
- 提供下载入口

历史不单独做复杂管理页，直接嵌在首页，便于使用。

### 6.3 响应式布局

桌面端：

- 左侧主区：输入 + 当前结果
- 右侧侧栏：最近历史

手机端：

- 上下布局
- 输入区在上
- 结果区在中
- 历史区在下

布局重点不是炫技，而是确保第一次使用也能立即明白怎么操作。

## 7. Gemini 调用设计

已验证中转站可用 OpenAI 兼容接口访问：

- `GET /v1/models`
- `POST /v1/chat/completions`

已验证 `gemini-3.1-flash-image` 可返回图片数据。

当前预期请求方式：

- 请求地址：`${GEMINI_PROXY_BASE_URL}/v1/chat/completions`
- 请求头：`Authorization: Bearer ${GEMINI_PROXY_API_KEY}`
- 内容类型：`application/json`

请求体核心结构：

```json
{
  "model": "gemini-3.1-flash-image",
  "messages": [
    {
      "role": "user",
      "content": "生成一张春天傍晚花园的写实照片，暖色调，自然光。"
    }
  ],
  "stream": false
}
```

当前已知响应关键字段：

```json
{
  "choices": [
    {
      "message": {
        "images": [
          {
            "type": "image_url",
            "image_url": {
              "url": "data:image/jpeg;base64,..."
            }
          }
        ]
      }
    }
  ]
}
```

服务端处理逻辑：

1. 调用接口
2. 读取 `choices[0].message.images[0].image_url.url`
3. 解析 `data:image/...;base64,...`
4. 提取 MIME 类型和 base64 内容
5. 解码为图片文件并保存到本地
6. 写入历史记录
7. 返回可展示的图片资源路径

## 8. 数据流

生成图片的数据流如下：

1. 用户在首页输入提示词并点击“生成图片”
2. 前端调用受保护的生成接口
3. 服务端验证登录状态
4. 服务端读取 `.env` 中的 `base_url`、`api_key`、`model`
5. 服务端请求 Gemini 中转站
6. 服务端接收图片 data URL
7. 服务端将图片落盘
8. 服务端写入 SQLite 历史记录
9. 前端接收最新结果并渲染
10. 首页历史区同步刷新

查看历史的数据流：

1. 页面加载时请求最近 30 天的历史记录
2. 服务端从 SQLite 查询并返回结果
3. 前端渲染缩略图、时间和摘要
4. 用户点击历史项后切换当前结果展示

## 9. 错误处理

前端错误提示要用普通人能理解的语言。

用户可见提示：

- 密码错误：`密码不正确`
- 网络或中转站问题：`生成失败，请稍后再试`
- 服务端配置异常：`服务配置异常，请联系维护者`
- 图片保存失败：`图片保存失败，请重试`

服务端日志保留技术细节，例如：

- 中转站返回的状态码
- 请求失败原因
- 图片解析失败原因
- 文件写入错误

前端不展示原始报错栈，避免暴露内部信息。

## 10. 30 天清理策略

历史记录保留 30 天。

清理逻辑分两层：

- 服务启动时执行一次清理
- 再通过定时任务每天执行一次清理

清理步骤：

1. 查询 30 天前的历史记录
2. 删除对应图片文件
3. 删除对应数据库记录
4. 记录清理结果日志

这样即使定时任务偶发漏跑，下次服务启动时也能兜底。

## 11. 文件与模块边界

建议模块拆分如下：

- 页面层：登录页、主页面
- 组件层：登录表单、提示词输入、结果卡片、历史列表
- 服务层：Gemini 请求封装、图片解析与保存、清理任务
- 数据层：SQLite 表定义与查询方法
- 鉴权层：session 与登录校验

模块边界原则：

- Gemini 调用逻辑不要散落在页面代码中
- 文件写入逻辑不要混入 UI 组件
- 数据库访问统一收口到数据层
- 历史清理逻辑单独放在服务模块中

## 12. 测试设计

至少覆盖以下验证点：

### 12.1 鉴权

- 正确密码可以登录
- 错误密码会被拒绝
- 未登录无法访问生成接口
- 未登录无法读取历史记录

### 12.2 Gemini 接入

- 能正确读取 `.env` 中的 `base_url`、`api_key`、`model`
- 能正确请求 `/v1/chat/completions`
- 能正确处理图片 data URL 响应

### 12.3 图片存储

- 成功生成后图片能落盘
- 返回 JPEG 时能正确写入文件
- 文件路径与数据库记录保持一致

### 12.4 历史记录

- 成功生成后能写入 SQLite
- 首页能按时间倒序读取历史
- 点击历史项能重新查看图片

### 12.5 清理任务

- 30 天前记录会被删除
- 对应图片文件会一起删除
- 不会误删 30 天内的新记录

## 13. 部署设计

部署目标是这台 VPS 直接提供服务。

建议部署结构：

- Next.js 应用进程
- SQLite 文件
- 本地图片目录
- Nginx 反向代理
- HTTPS

运行方式可选：

- `systemd`
- `pm2`

部署要求：

- `.env` 不进入公开仓库
- 图片目录具备写权限
- 定时清理任务稳定执行
- 反向代理正确转发到应用端口

## 14. 未来扩展空间

本次实现虽然聚焦极简版本，但保留以下扩展空间：

- 增加尺寸和张数选项
- 增加风格预设
- 增加失败重试
- 增加历史搜索和删除
- 更换中转站或模型时只调整 `.env`

这些扩展不影响当前设计的总体架构。

## 15. 结论

本项目采用服务端代理的单用户私有网站方案，以最少功能满足真实需求：

- 单密码登录
- 中文界面
- 首页完成生成、查看、下载、历史浏览
- Gemini 中转配置全部走 `.env`
- 图片本地落盘
- 历史记录保留 30 天并自动清理

该方案在当前 VPS 场景下实现成本低、维护简单、安全边界清晰，适合直接进入实施阶段。
